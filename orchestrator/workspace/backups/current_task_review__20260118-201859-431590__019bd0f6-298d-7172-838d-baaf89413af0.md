## 任务步骤（建议）
1. 阅读三份文档，逐份输出：关键术语、目标/非目标、数据流/调用流、与 project 相关的实体/接口/约束。
2. 在代码库中定位 project 相关实现：目录结构、主要类/函数/模块、入口点（API/CLI/任务）到 project 核心逻辑的调用链；给出文件与符号级证据（路径+行号，或搜索命令+关键匹配片段）。
3. 对照 KISS/DRY/SOLID/YAGNI，列出最主要的结构性问题（最多 10 条），每条附定位证据。
4. 给出“可执行的重构拆解建议”：目标目录结构、分步迁移路线（每步可独立验证）、建议新增/补齐的测试类型与范围（单测/契约/集成）。
5. 在报告中明确：哪些 dev_plan 任务可以被标记为 DONE/维持 TODO，以及依据是什么（以你提供的证据为准）。

## 输出要求（报告必须包含）
- 可复现命令：你运行过的 `rg`/`ls`/`tree`/测试命令等（含关键输出摘要）
- 关键文件清单：按“入口/领域/集成/基础设施”分类
- 建议更新 dev_plan 的具体文本（可直接复制进 dev_plan），包含 status/acceptance/evidence 字段

## 验收标准
- 覆盖三份文档要点，并明确与 project 重构的映射关系
- 能在代码中定位 project 相关模块与调用链，并给出可复现证据
- 给出可执行的重构任务拆解与测试策略，足以让 DEV/TEST 开始下一步工作
- TDD: 本工单为审阅取证，不直接写代码；因此不适用“先红后绿”。但你必须在拆解建议中明确哪些步骤需要先补测试（红-绿-重构）以及对应测试位置/命令。
